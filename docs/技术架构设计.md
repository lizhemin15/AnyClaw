# AnyClaw 技术架构设计与选型

## 1. 总体架构

```
                    ┌─────────────────────────────────────────────────────────┐
                    │                      用户 / 浏览器                        │
                    └───────────────────────────┬─────────────────────────────┘
                                                │ HTTPS
                    ┌───────────────────────────▼─────────────────────────────┐
                    │  AnyClaw-Web (前端)                                        │
                    │  登录 / 注册 / 购买 / 控制台 / 与 Claw 交互                   │
                    └───────────────────────────┬─────────────────────────────┘
                                                │ REST / WebSocket
                    ┌───────────────────────────▼─────────────────────────────┐
                    │  AnyClaw-API (业务后端)                                   │
                    │  鉴权 / 订单 / 容器生命周期 / 路由到用户 Claw 实例            │
                    └───────┬─────────────────────────────┬───────────────────┘
                            │                             │
              ┌─────────────▼─────────────┐   ┌───────────▼───────────┐
              │  数据库 (用户/订单/实例)   │   │  容器编排 / 调度层      │
              │  PostgreSQL / SQLite     │   │  Docker API / 自研     │
              └──────────────────────────┘   └───────────┬───────────┘
                                                         │
              ┌──────────────────────────────────────────▼──────────────────────┐
              │  宿主机池 (多台 2GB 轻量服务器)                                    │
              │  ┌─────────┐ ┌─────────┐     ┌─────────┐                         │
              │  │ 容器 A  │ │ 容器 B  │ ... │ 容器 N  │  ← 每台约 100+ 实例      │
              │  │ AnyClaw │ │ AnyClaw │     │ AnyClaw │                         │
              │  └─────────┘ └─────────┘     └─────────┘                         │
              └──────────────────────────────────────────────────────────────────┘
```

## 2. 各模块选型与职责

### 2.1 AnyClaw-Server（已有）

- **技术**：Go，现有 GoReleaser + GitHub Actions 已支持多架构 Docker 镜像构建与发布。
- **职责**：提供可部署的 Open Claw 精简镜像（AnyClaw）；通过 GitHub Action 完成编译与镜像推送，无需本地编译环境。
- **建议**：在 release 流程中增加「AnyClaw」品牌镜像 tag（如 `anyclaw/anyclaw:latest`），与现有 openclaw 镜像可并存或作为别名。

### 2.2 AnyClaw-Web（前端）

| 项目     | 选型           | 说明 |
|----------|----------------|------|
| 框架     | **Next.js (App Router)** 或 **React + Vite** | 需 SEO 用 Next；纯控制台可 Vite，打包小、迭代快。 |
| 语言     | TypeScript     | 与 API 契约一致，减少联调错误。 |
| UI       | **Tailwind CSS + shadcn/ui** 或 **Ant Design** | 快速做登录/注册/商品/控制台；shadcn 更轻、可定制。 |
| 状态     | React Query + Zustand | 服务端状态用 React Query，全局 UI 状态用 Zustand。 |
| 鉴权     | JWT 存 httpOnly Cookie 或内存 + 刷新 Token | 由 API 统一发 Token，前端只带 Cookie 或 Header。 |

**功能范围**：注册、登录、套餐展示与购买、个人实例列表、单实例控制台（与 AnyClaw 交互的聊天/指令界面）、基础设置与账单；**管理员**：节点管理（列表/添加/编辑/禁用）、一键部署与更新、修改节点配置（如 max_instances）、查看实例分布与简单监控。

### 2.3 AnyClaw-API（后端）

| 项目     | 选型           | 说明 |
|----------|----------------|------|
| 语言/框架| **Go + Gin/Fiber** 或 **Node.js + Fastify** | Go 与 AnyClaw-Server 同栈，部署简单；Node 生态熟可选 Fastify。 |
| 数据库   | **PostgreSQL**（生产） / **SQLite**（开发/极小规模） | 用户、订单、实例绑定、配额均落库；生产建议 PostgreSQL。 |
| ORM/访问 | **sqlc** 或 **GORM**（Go）/ **Drizzle** 或 **Prisma**（Node） | 优先 sqlc：类型安全、性能好、与 Go 契合。 |
| 鉴权     | JWT（access + refresh）或 Session | 登录后发 JWT；API 校验后解析 userId，再查实例归属。 |
| 支付     | ** Stripe** / **支付宝/微信（国内）** | 支付回调后写订单状态，再触发「开通实例」流程。 |
| 容器调度 | 见下节「容器与调度设计」 | API 只发「创建/停用/删除」等指令，由调度层执行。 |

**核心接口**：

- 认证：`POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`
- 用户：`GET /me`, `PATCH /me`
- 商品/订单：`GET /plans`, `POST /orders`, `GET /orders/:id`，支付回调 `POST /webhooks/payment`
- 实例：`GET /instances`, `POST /instances`（付费成功后内部或异步调用）, `GET /instances/:id`, `DELETE /instances/:id`
- 与 Claw 交互：`GET /instances/:id/ws` 或 `POST /instances/:id/chat` 等，由 API 根据 instance 找到对应容器 WebSocket 并做双向转发。
- **管理端（仅管理员）**：`GET/POST /admin/nodes`（节点列表、添加节点）, `PATCH /admin/nodes/:id`（修改 max_instances、状态）, `POST /admin/nodes/:id/deploy` 或 `POST /admin/deploy`（一键部署/更新所选节点）, `GET /admin/stats`（实例分布、节点心跳等）。

### 2.4 容器与调度设计（灵活调度）

目标：一台 2GB 双核机器跑尽量多 AnyClaw 容器（约 10MB RAM/个），且支持多机、故障转移与弹性伸缩。

| 层级       | 选型与思路 |
|------------|------------|
| 单机运行时 | **Docker**（已有镜像），不引入 K8s 以节省资源。 |
| 调度中心   | **自研轻量调度服务**（与 API 同机或独立均可）：维护「实例 ID ↔ 宿主机 + 容器 ID」映射，决定在哪台机器创建/迁移。 |
| 通信方式   | API 通过 **SSH** 或 **Docker HTTP API（TCP + TLS）」** 调用宿主机上的 Docker，执行 create/start/stop/rm。 |
| 发现与路由 | 容器通过 WebSocket 主动连 API 并注册；API 维护 `instance_id -> 容器 WebSocket` 映射，无需记录容器 host/port，也不为容器暴露端口。 |
| 弹性与故障 | 宿主机心跳上报「剩余内存/容器数」；新实例优先选剩余容量最大的机器；某机失联则将该机实例标记为「待迁移」，由调度在其它机器重建并更新映射。 |

**简化实现（MVP）**：

- 单台 2GB 服务器：仅跑「AnyClaw-API + 调度逻辑 + Docker + 所有 AnyClaw 容器」。
- 调度逻辑内嵌在 API 或独立进程：收到「开通实例」→ 选本机 → `docker run -d --memory=12m --name=anyclaw-{userId}-{instanceId} ...` → 写 DB 与内存映射。
- 用户与 Claw 交互：容器主动连 API 的 WebSocket 并注册；API 根据 instanceId 找到该实例的 WebSocket 连接并做双向转发，无需容器端口映射。

**扩展多机**：

- 调度服务维护「宿主机列表 + 每机剩余可创建数」；创建时选一台有余量的机器，通过该机 Docker API 创建容器；DB 存 `(instance_id, host, container_id)` 即可，无需 port（容器不暴露端口，通过 WebSocket 连 API）。
- 可选：每台宿主机跑一个 **Agent**，负责本机 Docker 操作与上报状态，API 只和 Agent 通信，便于以后加安全与限流。

### 2.5 用户与 AnyClaw 的「交互」方式（容器不暴露端口）

采用 **容器主动 WebSocket 连 API**，容器对外零端口：

- **连接方向**：AnyClaw 容器启动后，作为**客户端**主动连接 AnyClaw-API 的 WebSocket 端点（如 `wss://api.anyclaw.com/containers/connect`），携带 `instance_id` 与启动时注入的 `token`；API 校验后将该连接与实例绑定。
- **数据流**：用户 ↔ 浏览器 WebSocket ↔ AnyClaw-API ↔ **已绑定的容器 WebSocket** ↔ AnyClaw。API 只做「按 instance_id 查连接 + 双向转发」，不解析业务载荷。
- **效果**：容器无需监听端口、无需暴露到公网或内网端口，NAT/防火墙友好，安全与部署更简单。

**性能评估（对 API 的要求）**：

| 维度 | 评估 | 说明 |
|------|------|------|
| 连接数 | **压力不大** | 每容器 1 条长连；单机 100 容器 = 100 条；API 单进程可轻松支撑数千条 WebSocket（Go/Node 默认即可）。 |
| 内存 | **很低** | 每条连接约 4～64KB 缓冲；300 条约 10～20MB，相对 2GB 宿主机可忽略。 |
| CPU | **很低** | 仅做「收包 → 查表 → 转发」，无业务解析与重计算；瓶颈在容器内 LLM/工具，不在 API。 |
| 对比「API 连容器」 | **更省** | 无需为每个容器分配端口、无需 API 主动建连，只需维持已建好的反向连接并做路由。 |

**结论**：该方案对 AnyClaw-API 的性能要求不高，API 主要承担「连接保持 + 消息路由」；重算在容器内，扩展时只需保证 API 能支撑「实例总数」量级的常驻 WebSocket 即可。

**AnyClaw-Server 侧改造**：需增加「出站桥接」能力：容器内进程启动后根据环境变量（如 `ANYCLAW_API_URL`、`ANYCLAW_INSTANCE_ID`、`ANYCLAW_TOKEN`）主动连接 API 的 WebSocket，注册实例并复用现有 Pico 协议或约定好的 JSON 帧做双向转发；可新增一个轻量 channel（如 `anyclaw_bridge`）或独立小模块，在 gateway 中启用后不再监听端口，仅维持该出站连接。

### 2.6 编译与 CI/CD（GitHub Actions）

- **现状**：AnyClaw-Server 已有 `release.yml`（打 tag → GoReleaser → 多架构二进制 + Docker 镜像）、`docker-build.yml`（workflow_call 构建并推 GHCR/Docker Hub）。
- **建议**：
  - 保留现有流程；如需「AnyClaw」独立镜像名，在 GoReleaser 的 `dockers_v2` 中增加一条或复用现有镜像多加 tag（如 `anyclaw/anyclaw:{{ .Tag }}`）。
  - 若 AnyClaw 与 openclaw 同仓库同二进制：只增加镜像仓库/标签即可。
  - AnyClaw-API / AnyClaw-Web：各自仓库或同 monorepo 下用 GitHub Actions 做 lint、test、build 与部署（构建产物推云主机或通过 SSH 拉取镜像/二进制）。

## 3. 地域与部署架构（大陆 + 香港，共用云数据库）

### 3.1 整体布局

| 区域 | 用途 | 部署内容 |
|------|------|----------|
| **大陆 1 台主服务器** | 国内用户入口 | AnyClaw-API + AnyClaw-Web |
| **香港 1 台主服务器** | 海外/香港用户入口 | AnyClaw-API + AnyClaw-Web |
| **香港多台节点服务器** | 跑 AnyClaw 容器 | 仅部署 **Agent + Docker**，不跑 API/Web；可随时增加机器做扩容 |
| **云数据库（1 套）** | 统一数据源 | 托管 PostgreSQL（如阿里云 RDS / 腾讯云 CDB）；大陆 API 与香港 API **共用同一实例**，保证用户、订单、实例、节点数据一致 |

```
                    大陆用户 ──► 大陆主服务器 (API + Web)
                                        │
                    海外用户 ──► 香港主服务器 (API + Web)    共用云数据库 (PostgreSQL)
                                        │                              ▲
                                        │ 调度 / 下发任务                │
                                        ▼                              │
                    香港节点1 (Agent + Docker) ── 容器 WebSocket 连 API ─┘
                    香港节点2 (Agent + Docker) ── ...
                    香港节点N (Agent + Docker) ── 管理员可一键扩容、部署、修改
```

流量与数据流：

- 用户按地域或解析访问：大陆用户 → 大陆 API/Web；海外用户 → 香港 API/Web。
- 两台 API 都连**同一套云数据库**，读写同一批表；调度、实例归属、管理员操作均以 DB 为准。
- 仅**香港节点**上运行 AnyClaw 容器；大陆主服务器不跑容器。容器通过 WebSocket 连**当前区域**的 API（大陆用户实例连大陆 API，香港用户实例连香港 API；若希望所有容器统一连香港 API 也可在调度时约定）。

### 3.2 节点与 Agent

- 每台「香港节点」服务器上部署轻量 **Agent**：负责本机 Docker 的创建/停止/删除、上报本机状态（已用实例数、剩余可创建数、心跳）。
- API 不直连节点上的 Docker，只与 Agent 通信（HTTPS 或带 token 的 WebSocket），由 Agent 执行 `docker run` 等。
- **节点表**（在共用 DB）：如 `hosts`(id, name, region, agent_url 或 ssh_info, token_hash, max_instances, current_instances, status, last_heartbeat, created_at)。调度时只选 `region=HK` 且 `current_instances < max_instances` 且心跳正常的节点。

### 3.3 管理员前端与一键运维

所有**扩容、部署、修改**均通过**前端管理员账号**在管理后台完成，无需登服务器手改。

| 能力 | 实现方式 |
|------|----------|
| **一键扩容** | 管理员在「节点管理」中点击「添加节点」，填写新香港服务器的 Agent 地址 + 认证 token（或 SSH 信息，由 API 自动安装/注册 Agent）；保存后写入 DB，该节点即参与调度，新实例可分配上去。 |
| **一键部署 / 更新** | 管理员选择「全部节点」或勾选部分节点，点击「部署 / 更新」；API 向这些节点的 Agent 下发指令（如拉取最新镜像 `anyclaw/anyclaw:latest`、重启容器或仅更新后续新起容器使用的镜像）；Agent 执行后上报结果，前端展示成功/失败。 |
| **修改配置** | 节点列表展示每台机器的 max_instances、当前实例数、状态；管理员可修改某节点的 max_instances、禁用/启用节点；修改写回 DB，调度立即生效。 |
| **查看与运维** | 节点列表、实例分布、简单监控（心跳、失败任务）均在管理后台展示；可选告警（某节点失联、实例创建失败）由 API 轮询或 Agent 上报后写表，前端展示。 |

实现要点：

- **鉴权**：管理端接口（如 `GET/POST /admin/nodes`、`POST /admin/nodes/:id/deploy`）仅对**管理员角色**开放（JWT 中 role=admin 或单独 admin 表/权限中间件）。
- **Agent 与 API 的协议**：约定 REST 或 WebSocket（如 `POST /agent/run` 传 instance_id、image、env；`GET /agent/status` 返回本机负载）；API 调用 Agent 时带节点 token 或签名，防止伪造。
- **一键部署**：可封装为「下发“拉取镜像 + 可选重启”」的批量任务，Agent 执行后回调 API 或由 API 轮询 Agent 状态；前端显示进度与结果。

## 4. 数据模型（核心）

- **users**：id, email, password_hash, role(user|admin), created_at, plan_id 等。
- **plans**：id, name, price, duration_days, max_instances 等。
- **orders**：id, user_id, plan_id, amount, status, paid_at, external_id（支付单号）等。
- **instances**：id, user_id, name, host_id（关联节点）, container_id, status(creating/running/stopped/error), created_at 等（无需 port，容器经 WebSocket 连 API）。
- **hosts**（节点表）：id, name, region(hk), agent_url 或 ssh_info, token_hash, max_instances, current_instances, status(enabled|disabled|error), last_heartbeat, created_at；供调度与管理员前端使用。

---

## 5. 安全要点

- 所有对外接口鉴权；实例相关接口校验 `instance.user_id == current_user_id`。
- 支付回调校验签名，防伪造。
- 宿主机 Docker API 不暴露公网；仅 API/调度服务通过内网或 SSH 访问。
- 用户与 Claw 的通信经 API 转发，避免容器直接暴露。

## 6. 技术栈汇总

| 模块           | 选型 |
|----------------|------|
| 前端           | Next.js 或 React+Vite, TypeScript, Tailwind, shadcn/ui（含管理员后台） |
| 后端 API       | Go (Gin/Fiber) 或 Node (Fastify), JWT, 双区部署（大陆 + 香港） |
| 数据库         | **共用一套云数据库**（托管 PostgreSQL，大陆/香港 API 同连） |
| 支付           | Stripe / 支付宝·微信 |
| 容器节点       | 香港多台；每台部署 Agent + Docker，不暴露容器端口，容器 WebSocket 连 API |
| 调度           | 自研（内嵌 API），通过 Agent 在节点上执行创建/停止；节点表在共用 DB |
| 编译/发布      | GitHub Actions + GoReleaser（已有），API/Web 单独 workflow |

此架构支持大陆/香港双入口、香港节点池弹性扩容，并通过管理员前端实现一键扩容、部署与修改。
